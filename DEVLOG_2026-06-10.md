# Collie Development Log — 2026-06-10

## Strategic Shift: Extraction Engine → Analyst System

### Problem
The prototype had grown accurate at *extracting* data but still behaved like an
extraction engine rather than a real estate analyst:

- Too much reliance on cell proximity and extraction heuristics.
- GPT acting as a parser, not an analyst.
- Limited transparency into how a fact was selected.
- No mechanism for accumulating institutional knowledge across runs.
- Users couldn't easily see what the system reviewed, trusted, rejected, or why.

### Decision
Formalize the pipeline as an analyst workflow, with the **Verified SSOT** and
**Analyst Bundle** as the primary outputs and the memo as secondary:

```text
Upload
  ↓
Workbook Mapper        (sheet_classifier.py — content-role classification)
  ↓
Section Reader         (section_reader.py — authoritative-tab reading)
  ↓
Catalog Validation     (metric_resolver.py — bounds / source hierarchy)
  ↓
Reconciliation         (tools._reconcile_bounded_metrics — backfill + derive)
  ↓
Final Identity Checks  (tools.py:902 — run ONCE, on final values)
  ↓
Verified SSOT          (ssot.py)
  ↓
Analyst Bundle         (analyst_bundle.py)
  ↓
Business-Plan / Memo   (scenarios/*)
```

---

## 1. Workbook Understanding

**Problem.** The engine could still source a value from an arbitrary workbook
location. A human analyst instead works top-down: Summary → Inputs → Cash Flow →
Debt → Returns.

**Changes.**
- Content-role classification (`sheet_classifier.py`) — authority decided by row
  labels, not sheet name. (A "Charts" tab whose content reads as a summary is
  still nominated; name blacklisting was explicitly reverted.)
- Authoritative-tab nomination with per-role caps + `_catch_all` fallback.
- Source-hierarchy enforcement per metric (`source_primary` / `source_forbidden`
  by sheet **role**).

**Outcome.** The SSOT and Analyst Bundle now record which sheets were used,
which were considered authoritative, and which were skipped.

## 2. Retail Acquisition Regression Fixes

Testing on *Retail-Acquisition Underwriting.xlsx* (sheets: Assumption,
Proforma, Cash Flow, Lease Roll Over, Rent Roll, OPEX, T12, SOFR Curve) exposed
several gaps. All fixes verified in code.

### Issue 1 — Hold Period not detected
- **Observed:** `Hold Period = —`. Actual: `Assumption!B4 = 60` (months).
- **Fix:** month→year normalization in `tools.py:676`. If a hold-period value is
  `> 24` and `<= 360`, treat as months and divide by 12. Logged as
  `Pattern fired: hold_period_gt_24_means_months`.
- **Outcome:** `60 months → 5.0 years`.

### Issue 2 — Exit Date not derived
- **Observed:** Acquisition Date present, Exit Date missing.
- **Fix:** `tools.py:697` derives Exit Date (and conversely Hold Period from
  Purchase + Exit dates), stored with `status="derived"`,
  `source_cell="Exit Date - Purchase Date"`.
- **Outcome:** Exit Date generated automatically from verified facts.

### Issue 3 — Debt assumptions not found
- **Observed:** `Debt / LTV / Interest Rate = —`. The model stores debt inside
  the **Assumption** tab (`B22 LTV=0.6`, `B25 Total Debt`, `B28 Term=36`,
  `B31 Rate Spread=0.035`), not a dedicated Debt tab.
- **Fix:** expanded source hierarchy so debt metrics accept `['debt','inputs',
  'summary']` as primary (verified for Debt Amount, Original/Current LTV, Debt
  Yield, DSCR, etc.).
- **Outcome:** debt metrics found even when no debt-classified sheet exists.

### Issue 4 — Monthly NOI presented as annual
- **Observed:** user asked for annual NOI trajectory; monthly rows returned.
- **Fix:** `flexible_extractor._annualize_if_monthly` (line 1002) groups monthly
  columns by year and sums (`sum_monthly_columns_by_year`); `agent_loop.py:129`
  instructs the agent to annualize monthly tables and label them as such.
- **Outcome:** annual NOI reflects yearly totals, clearly labeled as annualized.

### Issue 5 — Missing periodicity metadata
- **Fix:** `flexible_extractor._detect_periodicity` (line 980) tags series
  monthly/annual so downstream consumers don't conflate them.

### Issue 6 — Identity checks running before reconciliation
- **Problem:** reconciliation could change values *after* validation, so identity
  checks ran on stale numbers.
- **Fix:** identity checks removed from the cached extraction path
  (`tools.py:566`) and now run **once, after** reconciliation/backfill/derivation
  (`tools.py:902`). Cap Rate ≈ NOI/Price, LTV ≈ Debt/Cost, Debt+Equity ≈ TPC are
  all checked on final values.

## 3. Analyst Bundle (the engine→trust bridge)

**Problem.** The system produced facts but didn't explain what it looked at,
trusted, rejected, and why.

**Changes (`analyst_bundle.py`, surfaced in `app.py`).** The bundle now packages:
- **Workbook Map** — content roles, authoritative tabs, skipped sheets.
- **Extraction Plan** — metric → expected source role → actual source used.
- **Verified Facts** — value, `sheet!cell` provenance, method, status.
- **Issues / QC flags** — missing / suspicious / conflict / candidate-pool /
  fallback usage.
- **Identity Checks** — pass/fail on final values.
- **Source Audit** — accepted vs rejected candidates per metric.
- **Knowledge Usage** — which active patterns fired.
- **Run Metadata** — file, timestamp, version constants.

It writes per-run JSON to `assets/analyst_bundles/` (gitignored). Status split:
`{verified, derived, inferred, not_applicable}` = facts;
`{missing, suspicious, conflict, candidate_pool}` = review.

**Outcome.** The bundle is the primary trust/debugging artifact — it localizes
whether a remaining problem is classification, section reading, units,
validation, or memo.

---

## 4. Knowledge Layer (institutional memory)

**Problem.** Each upload was *analyze → forget*; the system didn't improve from
repeated testing.

**Architecture (`knowledge_store.py`, `knowledge/`).** Three strictly separated
concepts:

- **Observations** (`knowledge/observations/`) — deal-specific reviewed
  evidence. **Never** loaded by runtime extraction or prompt construction.
- **Patterns** (`knowledge/patterns/{model,metric,business_plan}_patterns.json`)
  — distilled reusable knowledge. Only `status: "active"` entries influence
  runtime; `candidate/draft/rejected/superseded/archived/inactive` are ignored.
- **Runtime block** — a small prompt fragment built from active patterns only,
  scoped to `{workbook_mapping, metric_resolution, validation, business_plan,
  chat}`.

Promotion lifecycle: **Observation → Hypothesis → Rule**, with human approval
represented by an `active` status. GPT may *propose* an observation but never
decides truth or creates a runtime rule automatically. (All four seeded patterns
are currently `candidate` — none yet active — so runtime behavior is unchanged
until a human promotes them.)

**Outcome.** A safe mechanism to accumulate institutional knowledge without
letting GPT mint runtime rules.

---

## 5. GPT Role Redefined

- **Before:** GPT as extractor.
- **Now:** GPT as Workbook Reviewer / Fact Reviewer / Business-Plan Reviewer /
  Risk Reviewer — navigates structure, interprets the business plan, identifies
  risk, answers curveballs. It does **not** invent facts, override verified
  facts, or create rules.

## 6. Performance & Stability

**Problem.** Instability after adding Section Reader + Analyst Bundle +
Knowledge Layer (parallel GPT calls, OpenAI rate limits, heavy UI rendering).

**Changes.**
- Section Reader default model → `gpt-4o-mini` (`SECTION_MODEL`, env-overridable).
- Retry/backoff on section reads; sequential section processing to ease rate
  limits.
- Reduced GPT worker pool; trimmed Analyst Bundle payload; simplified
  source-audit rendering.

**Outcome.** Fewer hosted-deployment failures, more reliable runs.

# Part II — Two-Stage AAM Redesign (Human-in-the-Loop)

## Trigger
The analyst-system pipeline above was accurate but had two structural problems:

1. **Read amplification.** A single ingest opened each workbook **~9 times** and
   fired **7+ GPT calls** — 5 `section_reader` section sweeps + Pass 2 insight +
   comprehension review — all trying to land **all 109 catalog metrics** at once.
   Breadth was the cause of both the cost *and* the inaccuracy: no pass was ever
   told "find these specific fields and get them right."
2. **No human checkpoint.** The system ran analyze → memo with nothing in the
   middle. Facts were never confirmed by a person before the narrative was built.

## Decision
Split extraction into **two stages with a human verification gate between them**,
and make GPT **focused + on-demand** instead of broad + automatic:

```text
Upload
  ↓
STAGE 1 — Workbook Mapper (classify_sheets, GPT #1)
  ↓  role map
Focused AAM resolve (deterministic, scoped to ~20 fields)
  ↓
AAM blank-fill (one focused GPT call, GPT #2, on demand)
  ↓
📋 AUDIT APPENDIX  →  HUMAN VERIFIES / CORRECTS / CONFIRMS   ← the gate
  ↓
STAGE 2 — full ingest + apply verified values
  ↓
Formula trace from verified cells (data_only=False)
  ↓  fill checklist gaps
Merged Snapshot (paragraph + 2 business-plan bullets)
  ↓
Phase C — on-demand analyses + chat
```

The first pass now targets only the **Audit Appendix Metrics (AAM)** — do a few
things right, get them verified, *then* follow the model's own formulas outward.
This is how an analyst actually reads a model: find the anchor numbers, trace the
references to where the detail lives.

---

## Stage 1 — Focused AAM extraction + verification gate

### The AAM set (`aam.py`)
A curated **20-field / 6-group** set (Identity · Basis · Income · Debt · Timing ·
Return). Every id is an existing `in_bounded_list` catalog metric — AAM is a
**scope** over the catalog, not new definitions. Interest Rate is split into
floating-rate components (index/base + spread + cap). `validate_aam()` guards
against drift from the catalog.

### Focused extractor (`aam_extractor.py`)
`extract_aam()` reuses the existing engine, scoped:
- **Workbook Mapper** (`classify_sheets`, GPT #1) → content-role map.
- **Deterministic resolve** — `scan_workbook_for_candidates` + `resolve_metric`
  over the 20 AAM ids (schema bounds + source hierarchy intact).
- **`fill_aam_blanks()`** — one focused batched GPT call (GPT #2) over only the
  gaps, re-validated through `resolve_metric`.
- **`_normalize_aam()`** — representation fixes for the appendix: Hold Period
  months→years (active `hold_period_gt_24_means_months` rule); Equity Multiple
  rendered as `Nx` (the shared `ratio` formatter wrongly showed sub-1.5 as %).

Net: the first pass is **2 GPT calls** (map + focused read), down from 7+, and is
*more* accurate because each call is single-purpose.

### Audit Appendix + gate (`app.py`)
For **deal_review only**, an **additive, non-breaking** gate sits between upload
and analysis (`_render_aam_gate`, `_confirm_aam_and_ingest`):
- Editable appendix table (grouped, with `sheet!cell` provenance + status).
- First load is **deterministic-only** (fast/free); a **"🤖 Fill blanks with
  GPT"** button runs the focused gap-fill on demand (bulk-fill, then review).
- **Analysis is blocked until "✅ Confirm."** On confirm, verified/edited values
  are written back via `ssot.apply_verified_aam()` (status=`verified`,
  `human_verified=True`) — the human's confirmation is what approves them.

---

## Stage 2 — Formula trace + merged Snapshot + analyses

### Formula tracer (`formula_tracer.py`)
`trace_from_verified()` re-opens the workbook **with formulas** (`data_only=False`)
and does a bounded BFS from each verified AAM cell along its formula precedents
(openpyxl `Tokenizer` parses refs, incl. ranges + quoted cross-sheet names),
mapping landed cells → non-AAM catalog metrics via the alias index. Cross-sheet
refs are the goldmine: `NOI = ='Annual CFs'!G46` reaches EGI/OpEx; `Total
Project Cost = SUM(...)` reaches the equity stack.

### Trace → checklist integration (`ssot.merge_traced_metrics`)
Traced metrics are folded into `bounded_metrics` (**fill-only**, never clobbers a
verified value; schema-range guarded; canonical-name keyed) so **both the
Snapshot and the deep-dives** use them — not just a side field. Fixed a real bug:
the catalog has two "LTC" aliases (junk USD `ltc` vs ratio `loan_to_cost_ltc`);
a tracer matcher tiebreaker (`_metric_quality`, prefers ranged + checklist
metrics) now resolves to the right one.

### Merged Snapshot (`scenarios/deal_review.py`)
`generate_deal_review` now emits **one `## Snapshot` section** = a paragraph +
exactly 2 business-plan bullets. Risks and the old Audit-Appendix section are
dropped (the appendix *is* the gate now; risks is a Phase C analysis). The prompt
is told the facts are human-verified, and consumes a new FORMULA-TRACED METRICS
block (`_format_traced_metrics`).

### Phase C
After confirm, the 5 deep-dive analysis buttons (Capital Structure · Cash Flow ·
Return Profile · CapEx · Key Risks) + the chat input (freeform GPT prompt)
render — the "GPT prompt + select additional analyses" surface.

---

## What's deliberately deferred
**Retiring the old 9-read pipeline.** It still runs at confirm. Kept on purpose:
the tracer only reaches metrics *formula-linked* to AAM anchors, so deep-dive
metrics like occupancy, WALT, DSCR, and debt yield aren't reachable yet —
cutting the pipeline now would degrade coverage. Retirement (and the real
read-count win) comes once the tracer or a complementary focused reader covers
the full checklist.

from "broad sweep, hope it's right" to "nail the anchors, confirm, then trace."

# Part III — Review Fixes & Knowledge Wiring

A review of the Part II build surfaced four UX/correctness gaps plus one
material wiring gap in the knowledge layer. This pass closed them (the two
largest items — SSOT export/trust UX and the full learning-candidate pipeline —
were deliberately deferred to their own pass).

## 1. Knowledge layer was not wired into the hot paths
**Problem.** The JSON knowledge layer is sound (only `active` patterns reach
runtime; observations never do), but `build_runtime_knowledge_block()` was only
called by `sheet_classifier` (workbook_mapping) and `deal_review` (business_plan).
The **AAM extractor, blank-fill, GPT resolver, and fallback never injected active
rules** — so promoted patterns could not influence AAM extraction / trace /
return profile.

**Changes.**
- New `knowledge_store.with_active_rules(prompt, scopes)` — appends the active
  block to a system prompt, **no-op until a pattern is `active`** (so wiring it
  in changes nothing until a human promotes one).
- Injected into `aam_extractor._aam_gpt_call` (`metric_resolution` + `validation`
  — covers focused extraction AND blank-fill), `metric_resolver_gpt._gpt_pick`,
  and `metric_fallback._one_metric_gpt_call` (`metric_resolution`).
- Surfaced the already-existing `knowledge_diagnostics()` as a **🧠 JSON
  Knowledge** expander in the SSOT panel — `active / candidate / invalid` counts.

**Outcome.** Human-approved rules now actually reach the extraction prompts;
their state is visible. (Currently 2 active / 5 candidate / 0 invalid.)

## 2. Month/year normalization was too narrow
**Problem.** Hold Period only converted values `>24 and <=360`, so `12 / 18 / 24`
months were misread as years.

**Fix.** `aam_extractor._normalize_aam` now **date-cross-checks**: when Purchase
+ Exit dates exist (`_derived_hold_years`), `_interpret_hold_value` picks the
years-vs-months reading closest to the date span. Falls back to the `>24`
heuristic only when no dates exist.

**Outcome.** `24mo + 2y dates → 2.0y`; `12 + 12y dates → stays 12`; `60 + no
dates → 5.0y`. The ambiguous low values are now disambiguated by context.

## 3. Deep-dive metric lookup was exact-name only
**Problem.** `_bounded_pretty` did `bounded.get(name)` and printed `MISSING` on
any key mismatch — so a metric stored under a non-canonical key (e.g. Levered
IRR) could be silently dropped.

**Fix.** `deep_dives._resolve_bounded` + `_name_to_id` resolve a record by
canonical name OR alias/`metric_id`.

## 4. Confirm felt like one big jump
**Fix.** Staged into two deliberate steps: **"✅ Confirm verified facts"** →
a facts-locked summary (`N verified, M traced`) → a separate **"📝 Generate
Snapshot & analyses →"** button (`aam_analysis_requested` state +
`_render_post_confirm`). Analysis no longer auto-fires on confirm.

## Deferred (next pass)
- **SSOT export / review / trust UX** (beyond the current expander).
- **Learning-candidate pipeline**: user override → observation JSON → repeated
  observation → candidate pattern → human approval → active. Plus broader
  knowledge injection into snapshot / deep-dive prompts. The design principle
  holds: users create evidence; humans promote patterns; GPT never self-teaches.

**Validation.** All deterministic paths verified headless (compile clean +
streamlit `AppTest`): knowledge injection (910-char block injected), month
cross-check cases, alias lookup, and the staged confirm→generate flow. GPT-
dependent behavior (injected-rule effects, Snapshot prose) needs a live key run.
