# Part I — BAC Stage-1 Diagnosis & Extraction Fixes

## Trigger
First BAC export (post-GPT-fill) reviewed with notes. Symptoms: most facts
pulled from `Key UW Metrics` / `General Information` instead of the One Pager;
Total SF showed $192M (the purchase price) as "192,000,000 SF"; Hold Period = 1
(should be ~5); TPC = purchase price; NOI was the exit figure.

## Root causes & fixes

### 1. Tier 1 was a bucket, not a ranking
`SHEET_PRIORITY_TIERS` lumped one-pager / general-info / key-UW / summary into
one tier, and `ROLE_TO_TIER` put summary/inputs/sources_uses all at 1. Ties fell
through to label quality — and Key UW Metrics' clean atomic labels won every tie.

**Fix.** Split the hierarchy to match how an analyst reads:
- Tier 1: one-pager / investment summary / overview
- Tier 2: inputs / assumptions / sources & uses
- Tier 3: secondary summaries (general info, key UW metrics, "summary")
- Tiers 4–8: cash flow, capex, debt, returns, everything else

New `one_pager` content role (tier 1; `summary` demoted to 3) with a
`hierarchy_role()` equivalence shim (one_pager → summary) so the catalog's
source_primary lists (authored in the xlsx) still accept One Pager values.
`name_tier` recorded on every candidate as a resolver tiebreaker so the
one-pager NAME wins even when the classifier roles both sheets 'summary'.

### 2. Out-of-range GPT fills were shown instead of blanked
Total SF = 192M failed range validation but was kept as `suspicious` and
displayed. `_focused_gap_fill` only rejected `missing`.

**Fix.** Gap-fill now rejects `suspicious` results too — a blank the human can
fill beats a confident wrong number.

### 3. Hold Period had no sanity check against dates
`Inputs!M8 = 1` (a "Year 1" index) passed as 1 year despite Purchase→Exit dates
implying ~5. The date cross-check only chose months-vs-years; it never flagged
disagreement.

**Fix.** `_normalize_aam` now flags Hold Period `suspicious` when the value
disagrees with the date-derived span >20% under BOTH readings, with the derived
value suggested in the note. (Never overwrites — dates may be unverified.)
Verified on 425 Colorado: `12` vs ~3.1y span → flagged.

### 4. Single NOI field conflated going-in and exit
**Fix.** `exit_noi` (existing bounded catalog metric) added to the AAM Income
group. (Superseded by Part II's pricing derivation, which is the real fix.)

### 5. Gate table UX
Removed the redundant raw **Value** column and the **Period** column; one
editable column shows the formatted value (`$192.00M`, `8.17%`), parsed back
via `parse_numeric_value` with tolerance-based change detection so a rounded
display isn't mistaken for an edit and unchanged rows keep full precision.

# Part II — NOI Driven From Pricing (the identity governs)

## User reframe
> "Both going-in and exit NOIs should be driven from pricing. NOI/cap rate is
> the price."

Cell-matching NOI grabs whichever column it finds first (usually the higher
exit/stabilized figure). The cap-rate identity makes that moot:

- **Going-in NOI = Purchase Price × Going-in Cap Rate**
- **Exit NOI = Exit Value × Exit Cap Rate**

## Changes
- **Exit Value (`exit_value_terminal_value`) added to the AAM** as the exit-side
  pricing anchor → AAM is now **22 fields / 6 groups**.
- `aam_extractor._derive_noi_from_pricing()` runs at the end of `extract_aam`
  and after blank-fill. Derived NOIs get `status="derived"` and the FORMULA as
  their source (e.g. "Purchase Price × Going-in Cap Rate").
- Guards: derives only from pricing inputs that validated (a `suspicious`
  out-of-range cap is refused); the COMPUTED NOI is range-checked against its
  own schema (a bad input can't write a nonsense product); `skip_ids` lets a
  human-corrected NOI stand.
- **Confirm-time re-derivation** (`app._rederive_noi_from_verified`): runs just
  before `apply_verified_aam`, overlaying the human-verified gate values and
  re-deriving NOI — so correcting the cap/price/exit value at the gate flows
  through to the persisted SSOT. A directly-edited NOI is respected (skip).

**BAC validation:** Exit NOI = $242.16M × 7.50% = $18.16M (deterministic);
going-in NOI = 192M × 8.17% = $15.69M once the cap resolves — with a note
flagging the old $18.15M cell as the wrong column.

# Part III — Deal Analyzer (3-column frontend)

## Decision
Rebuilt the deal_review workspace per the user's wireframe: **process | Outcome
| Chat**. perf_vs_plan keeps the legacy single-column flow. Full build (layout
+ behaviors), chat as a form inside the column, landing kept.

```text
┌─ Input & analysis ─┬────────── Outcome ──────────┬──── Chat ────┐
│ upload             │ pre-confirm: editable AAM   │ transcript   │
│ progress bar+text  │   appendix (the gate)       │  └ Add to    │
│ Progress checklist │ post-confirm:               │    Outcome   │
│ Proceed → confirm  │   AAM Summary (verified)    │ suggested-   │
│ Run analysis       │   Snapshot                  │  analysis    │
│                    │   folded findings (✕)       │  chips       │
│                    │   Final Report              │ prompt form  │
│                    │                             │ Generate     │
│                    │                             │  Report→SSOT │
└────────────────────┴─────────────────────────────┴──────────────┘
```

- **Findings flow:** deep-dive results and chat answers land in the transcript;
  "Add to Outcome" folds them into the middle column (dedup by content
  signature, removable).
- **Generate Report → SSOT** (`_generate_report`): memorializes verified facts
  + snapshot + kept findings into one report via GPT (deterministic concat
  fallback) and persists it as `layers.underwriting.deal_report`. *Whatever
  ends up in the Outcome becomes the final SSOT.*
- Execution-order trick: the left column renders in two phases around the
  middle (Streamlit places output by column, not execution order) — extraction
  runs before the appendix builds; the Proceed button reads the edited
  appendix the middle returns.
- Removed dead code: `_render_aam_gate`, `_render_post_confirm`,
  `_render_deep_dive_buttons`.

New session keys: `outcome_findings`, `outcome_added`, `snapshot_md`,
`final_report_md`.

# Part IV — Stage-1 Strictness (second BAC export)

## Trigger
Second live BAC export: **Purchase Price = $88.00M from `Inputs!P43`** — wrong —
and going-in NOI = $7.19M (= 88M × 8.17%, the derivation working on a bad
input). User directive: *the first pass must read ONLY sheets identified as
summary / one pager / assumptions / inputs* — and read models the way they're
built: **inputs are hard-coded; everything else derives.**

## Diagnosis (workbook inspection)
- `Inputs!P43 = 88000` is the **Westin-only allocation** in an "ACQUISITION
  STATS" block (BAC = Westin + Aloft + Parking portfolio). Per-asset prices are
  hardcoded (`F23=88000, F24=63000, F25=41000`); the combined **$192M
  (`F26`) is a SUM formula**. `General Information!C11 = =Inputs!F26*1000`.
- The deterministic pass HAD $192M (`candidate_pool`) — **the GPT fill
  overwrote it**, because `candidate_pool` was treated as a fillable gap.
- Total SF "192,000 SF" / Location from `C11` were GPT fabrications — values
  that don't exist in the cells GPT cited.

## Four mechanisms (all in `aam_extractor`, version 2026-06-11.1)

1. **Stage-1 sheet whitelist** (`_stage1_context`, `_STAGE1_MAX_TIER = 3`):
   the deterministic scan AND the GPT-fill pair list are restricted to tier ≤ 3
   sheets (one-pager / inputs / assumptions / S&U / secondary summaries).
   Empty whitelist → fallback to unrestricted, logged. On BAC: One Pager,
   Inputs, WestinInputs, AloftInputs, General Information, Key UW Metrics.
2. **`_GAP_STATUSES = {missing, suspicious}`** — GPT fills holes; it never
   second-guesses a `candidate_pool` value the deterministic ranking already
   validated. THE $88M fix.
3. **Cell grounding** (`_grounded_value`): every GPT hit is verified against
   the real workbook — whitelisted sheet, cell exists, non-empty, and the
   claim matches the actual value (×1 / ×100 / ÷100 percent slack only —
   deliberately NO ×1000 slack, so "192,000" can't pass a 192,000,000 cell;
   containment for text; same-day for dates). The **actual cell value** is
   what gets used, never GPT's claim.
4. **Hardcode preference** (the modeling insight): a formula-cell map is built
   for whitelisted sheets (`data_only=False`, bounded window); candidates carry
   `is_hardcoded`; for `aam.AAM_HARDCODED_INPUTS` (identity, cap rates, LTV,
   rate/spread/cap, dates/hold) a hard-coded cell outranks a formula cell at
   equal sheet rank. **Deliberately excludes price/TPC/debt** — portfolio
   aggregates are formulas; preferring hardcodes there would grab the $88M
   slice. GPT pairs are tagged `[input]`; the prompt warns about per-asset
   slices in portfolio models.

Also: `_aam_source` now shows the derivation formula for derived NOIs (the
user couldn't tell how NOI was computed because the Source column showed "—").

## Validation
- **BAC:** Purchase Price $192.00M holds (and can no longer be overwritten);
  Exit NOI derives; debt fields are honest blanks (found on `Inputs` by the
  grounded fill on deployed).
- **Retail:** 9/22 from the single whitelisted `Assumption` tab — basis / debt /
  timing correct; returns blank (they live on the excluded Proforma). Accepted
  trade-off: blanks over off-scope grabs; recovery via grounded fill, Stage-2
  trace, and the human gate.
- **425 Colorado:** 17/22 from `Summary & Assumptions`; bogus hold flagged.
- Grounding unit tests: fabricated SF rejected, text-vs-numeric rejected,
  off-scope rejected, percent slack accepted, hardcode tiebreak correct.

# Part V — UI Polish (pre-QC pass)

Per screenshot feedback, before the user's full QC run:

- **Deep-dive buttons → suggested-prompt chips** in the Chat column, above the
  prompt box (teal pills via `st.container(key="dd_chips")` scoped CSS).
  Buttons, not `st.pills` — pills selection persists across reruns and would
  re-trigger the dive after `st.rerun()`.

  a NEVER-use-tables rule; post-confirm AAM Summary renders as grouped
  markdown bullets instead of a dataframe.

- **Progress bars in the left box:** AAM extraction and the confirm pipeline
  (ingest → apply verified → formula trace → table parse) each show
  `st.progress` with a plain-language description below, replacing the
  chat-message/st.status blocks.
- **Robot/human icons removed:** chat avatars hidden via CSS
  (`[data-testid^="stChatMessageAvatar"]`); 🤖 markers → "(GPT)" text suffix;
  emoji stripped from button labels.
- **Prompt-box mockup pass:** chips + prompt + Send merged into ONE bordered
  card (`st.container(border=True, key="chat_box")` wrapping a borderless
  form); "Suggested analyses" caption dropped; chip label "Cash Flow/NOI";
  full-width teal Send. **Square corners everywhere except buttons** — global
  CSS zeroes border-radius on containers, forms, expanders, chat messages,
  inputs/textareas, dropzones, dataframes; buttons/chips keep their rounding.
